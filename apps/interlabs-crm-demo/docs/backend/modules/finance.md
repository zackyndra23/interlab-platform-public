# Finance module

The **[Finance](../../business/system-overview.md#glossary-finance)** module owns four entities — **[PO](../../business/system-overview.md#glossary-po)** Customer, **[PR](../../business/system-overview.md#glossary-pr)** (purchase requisition), Invoice Manufacture, Invoice Customer — and drives two of the eleven master-PO stages: **Production** (via the **[PR PO-Out](../../business/system-overview.md#glossary-pr-po-out)** automation) and **Invoice** (via Invoice Customer issuance). All advances flow through `backend/src/services/po.service.js#advanceStatus`; this module never mutates `purchase_orders.current_status` directly.

The Finance router relaxes the per-creator listing scope: `[Finance]` users see every Finance row, like `[Superadmin, CEO]`, because three of the four tables are populated by upstream divisions (Sales submit, Sales submit, **[Technical](../../business/system-overview.md#glossary-technical)** **[BAST](../../business/system-overview.md#glossary-bast)** upload). See `backend/src/routes/finance.routes.js:16-45` for the rationale.

## Purpose

Finance is the financial-bookkeeping backbone of every customer **[PO](../../business/system-overview.md#glossary-po)**:

- **PO Customer** — Finance-side mirror of a Sales PO; auto-created at submit. Read/update only; deletion is allowed only while `workflow_status='registered'` (`finance.service.js:235-247`).
- **Purchase Requisition** — Finance-side mirror of a Sales PR; the `/upload-po-out` endpoint is the **PR PO-Out** automation — it advances the master PO from Processed to Production.
- **Invoice Manufacture** — Finance-initiated; tracks supplier billing and payment. Payment recording does **not** advance the master PO (`finance.service.js:601-603`).
- **Invoice Customer** — Draft auto-created when Technical uploads BAST; the `/upload-invoice` endpoint advances the master PO from BAST to Invoice.

The four-table separation matches **[MOD_finance](../../../interlabs-crm-demo/docs/MOD_finance.txt)** §FORM 1-4.

## Validators

Source: `backend/src/validators/finance.validators.js`. Joi-based, sharing primitives with the Sales validators so that auto-created Finance rows compose predictably:

- `uuid` — UUIDv4 only (`finance.validators.js:7`).
- `currency` — `IDR | USD | EUR` (`:8`).
- `amount` — non-negative `number().precision(2)`; `percent` is 0-100, also two decimals.
- `attachmentIds` — `Joi.array().items(uuid).min(1)` (used by trigger endpoints which require attachment proof); `optionalAttachmentIds` is the same with no min.
- `itemList` — array of `itemListEntry` objects with `item_name` required and unknown keys allowed (`:21-30`).

Trigger validators are the load-bearing ones because they enforce the **all three trigger fields together** rule from MOD_finance:

```js
// finance.validators.js:100-105 — PR PO-Out: po_out_number + po_out_date + attachments
const requisitionUploadPoOut = Joi.object({
    po_out_number: Joi.string().min(1).max(200).required(),
    po_out_date:   Joi.date().iso().required(),
    attachment_ids: attachmentIds.required(),
    note:          Joi.string().max(2000).allow('', null),
});
```

Symmetric trigger validators: `invoiceManufactureUploadPayment` (`:144-150`) requires `payment_date` + `payment_amount` + `attachment_ids`; `invoiceCustomerUploadInvoice` (`:188-193`) requires `invoice_number` + `attachment_ids`. List-query validators add status filters (`workflow_status` / `current_pr_status` / `payment_status` / `invoice_status`) on top of the shared `listQuery` (page/limit/search).

Update-shape validators (`poCustomerUpdate`, `requisitionUpdate`, `invoiceManufactureUpdate`, `invoiceCustomerUpdate`) all use `.min(1)` so an empty PUT body is rejected.

## Services

Source: `backend/src/services/finance.service.js` (902 lines). Internal layering:

- **Shared helpers** (`:53-129`): `listRows` for paginated `WHERE deleted_at IS NULL` queries, `requireRow` for soft-delete-aware lookups, and `attachFilesToEntity` which rebinds uploaded `file_attachments` rows to a `(related_module, related_entity_id)` tuple inside the trigger transaction. `attachFilesToEntity` throws `BadRequestError` if any attachment id fails to bind (`:123-127`) — that is the "attachment proof" check.
- **PO Customer** (`:135-247`): pure CRUD plus `createPoCustomerFromSalesPo` hook. The hook is called by `sales.service.submitSalesPo` inside its own transaction and copies fields 1:1 from the Sales PO row. `deletePoCustomer` enforces a state guard: only `workflow_status='registered'` rows are deletable (`:237-241`).
- **Purchase Requisition** (`:253-448`): CRUD + `createRequisitionFromSalesPr` hook + `processRequisition` (the PO-Out trigger). Updates are blocked once `current_pr_status='Processed'` (`:272-276`); same guard on delete.
- **Invoice Manufacture** (`:454-674`): full CRUD plus `recordInvoiceManufacturePayment`. The "registered" notification is emitted on the `null → present` transition of `invoice_number`, both in `createInvoiceManufacture` (`:511-522`) and `updateInvoiceManufacture` (`:578-592`). Once `payment_status='Paid'` the row is immutable and undeletable.
- **Invoice Customer** (`:680-880`): CRUD + `createInvoiceCustomerDraftFromBast` hook + `issueInvoiceCustomer` (the customer-invoice trigger). `Processed` is the immutable terminal state.

Trigger services run inside `db.withTransaction(async (c) => { ... })` and acquire `FOR UPDATE` row locks before mutating, so concurrent trigger calls on the same row serialize cleanly:

```js
// finance.service.js:413-419 — processRequisition advances master PO inside the trigger txn
const masterPo = await poService.advanceStatus(c, {
    poId: pr.related_po_id,
    newStatus: 'Production',
    actorUserId: actor.id,
    actorRole: actor.role,
    note: input.note || `PO Out ${input.po_out_number} issued to manufacturer`,
});
```

Every notification emitted from this module passes the same `c` client to `notificationService.emit`, so notifications either commit or roll back atomically with the domain mutation. This is the canonical pattern referenced from [`../notifications.md`](../notifications.md).

The hooks `createPoCustomerFromSalesPo`, `createRequisitionFromSalesPr`, `createInvoiceCustomerDraftFromBast` all assert that `client` was passed — they refuse to run outside a transaction (`:201`, `:315`, `:759`).

## DB tables

Source: `backend/migrations/007_finance_forms.sql`. All four tables have UUIDv4 PKs, `timestamptz` audit columns, soft-delete `deleted_at`, and FK `created_by` / `updated_by` to `users(id) ON DELETE SET NULL`. Currency defaults to `'IDR'` everywhere.

| Table | Key columns | Constraints | Forward FKs to other modules |
|---|---|---|---|
| `po_customer_records` | `po_customer_record_number` (UNIQUE), `related_sales_po_id`, `related_po_id`, `customer_id`, `current_po_status` (mirror), `workflow_status` | `po_customer_workflow_chk` ∈ `{registered, active, invoiced, completed}` | `sales_purchase_orders`, `purchase_orders`, `customers`, `quotations` |
| `purchase_requisitions` | `pr_record_number` (UNIQUE), `related_sales_pr_id`, `related_po_id`, `related_po_customer_id`, `po_out_number`, `po_out_date`, `current_pr_status` | `pr_finance_status_chk` ∈ `{Registered, Processed}` | `purchase_requests_sales`, `purchase_orders`, `po_customer_records`, `customers` |
| `invoice_manufactures` | `invoice_manufacture_record_number` (UNIQUE), `related_pr_id`, `related_po_out_number`, `related_po_id`, `invoice_number`, `payment_date`, `payment_status` | `invoice_manu_payment_status_chk` ∈ `{Unpaid, Paid}` | `purchase_requisitions`, `purchase_orders` |
| `invoice_customers` | `invoice_customer_record_number` (UNIQUE), `related_po_customer_id`, `related_bast_id`, `related_do_id`, `related_po_id`, `invoice_number`, `invoice_status` | `invoice_customer_status_chk` ∈ `{Registered, Processed}` | `po_customer_records`, `bast_records` (FK added in migration `008`), `delivery_orders`, `purchase_orders` |

Note the deliberate FK staging on `invoice_customers.related_bast_id`: column declared `NULL` in `007:141`, FK added later in `008_*.sql` once `bast_records` exists. This is the only forward-reference shim in the Finance schema.

`workflow_status` on `po_customer_records` is the workflow mirror that `po.service.advanceStatus` syncs via `syncPoCustomerForMasterPo` (see [`../po-state-machine.md`](../po-state-machine.md)) — Finance code never writes it directly.

File attachments flow through the `file_attachments` table with `related_module` strings: `'finance.purchase_requisitions'`, `'finance.invoice_manufactures'`, `'finance.invoice_customers'`. PO Customer attachments use `'finance.po_customer_records'` if/when added; the schema currently relies on `attachFilesToEntity` to bind them.

## Automations

Finance is on both sides of the master-PO automation graph: it consumes one upstream automation (BAST → Invoice draft) and fires two downstream automations (PR PO-Out → Production, Invoice Customer issued → Invoice).

### Incoming: BAST signed → Invoice Customer draft

When Technical uploads a BAST (or PM completion BASTP, or explicit `sendBastToFinance`), `technical.service.runInstallationAutomation` reaches into `financeService.createInvoiceCustomerDraftFromBast` (`finance.service.js:755-795`) inside its own transaction. The hook inserts a draft `invoice_customers` row at `invoice_status='Registered'` with monetary fields blank (Finance fills them on `/upload-invoice`) and emits `finance.invoice_customer.registered`. The master-PO advance to BAST happens in the Technical service, not here. Full trigger walkthrough: [`../po-state-machine.md#automation-bast-signed--invoice-draft-created`](../po-state-machine.md#automation-bast-signed--invoice-draft-created).

### Outgoing: PR PO-Out → Production

`processRequisition` (`finance.service.js:374-434`) is the only path that flips `purchase_requisitions.current_pr_status` to `Processed`. The trigger inputs are validated by `requisitionUploadPoOut` (all three required together). Inside the transaction it: (1) binds attachments via `attachFilesToEntity`, (2) writes `po_out_number` + `po_out_date` + `current_pr_status='Processed'`, (3) calls `poService.advanceStatus({newStatus:'Production'})` which writes the four side-effects (history, tracking, `finance.po.production`, status update) and syncs `po_customer_records.workflow_status='active'`, (4) emits `finance.pr.processed` on top. Full automation contract: [`../po-state-machine.md#automation-pr-purchase-requisition-marked-po-out--production`](../po-state-machine.md#automation-pr-purchase-requisition-marked-po-out--production).

State-guard rejections: re-fire on a `Processed` PR throws `ConflictError` (`:387-389`); a PR without `related_po_id` throws `BadRequestError` (`:390-394`). Both surface as 4xx to the client without partial mutation.

### Outgoing: Invoice Customer issued → Invoice

`issueInvoiceCustomer` (`finance.service.js:808-868`) is the only path that flips `invoice_customers.invoice_status` to `Processed`. Trigger inputs validated by `invoiceCustomerUploadInvoice` (`invoice_number` + `attachment_ids` required). Inside the transaction: (1) bind attachments, (2) persist `invoice_number` + `invoice_date` + `invoice_status='Processed'`, (3) call `poService.advanceStatus({newStatus:'Invoice'})` which fires `finance.po.invoice` and flips `po_customer_records.workflow_status='invoiced'`, (4) emit `finance.invoice_customer.processed`. Full contract: [`../po-state-machine.md#automation-invoice-customer-issued--invoice-stage`](../po-state-machine.md#automation-invoice-customer-issued--invoice-stage).

State-guard rejections: re-fire on a `Processed` row throws `ConflictError` (`:819-823`); missing `related_po_id` throws `BadRequestError` (`:824-828`).

### Non-automation: Invoice Manufacture payment

Recording payment on an Invoice Manufacture (`recordInvoiceManufacturePayment`, `:605-662`) deliberately does **not** advance the master PO — supplier payment is a Finance-internal financial event that the customer-facing PO lifecycle ignores (MOD_finance §FORM 3 Trigger 2). It does emit `finance.invoice_manufacture.paid` and lock the row (`payment_status='Paid'` → immutable, undeletable).

## SLA hooks

Finance owns no SLA jobs of its own. The four registered jobs in `JOB_DEFINITIONS` (`backend/src/jobs/scheduler.js:37-70`) belong to other divisions; see [`../jobs.md`](../jobs.md) for the full catalogue. Finance is a recipient on two of them, plus a future SLA escalation:

- **Sales PO 2-working-day deadline** (per CLAUDE.md line 71, not yet implemented per [`../jobs.md#not-yet-implemented`](../jobs.md#not-yet-implemented)): when added, escalations target `[Superadmin, CEO, Admin & Log, Finance]`. Finance will appear in the recipient list as a passive observer, not as the SLA owner.
- **Technical PO 30-day due-date reminder** (`technical_po_due_reminder`, daily 08:00 WIB, `slaPoDueDate.job.js`): the engineer + Technical team are the primary recipients; Finance is included indirectly via standard `finance.po.production` recipient defaults but does not get a dedicated escalation in the current job (see `notifications.md` for `technical.po.due_date_reminder` recipient set).

If a `sla_finance_*` job is ever added — for example, an "Invoice Customer Registered older than X days" reminder — it should follow the patterns in [`../jobs.md#extension-points`](../jobs.md#extension-points): register in `JOB_DEFINITIONS`, run in `Asia/Jakarta`, emit a new template via `notificationService.emit`, never mutate `purchase_orders.current_status` directly.
